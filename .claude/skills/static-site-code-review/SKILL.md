# 静的サイト コードレビュー手順

このSkillは、TypeScript/ビルドツールを使わない単一HTMLファイル構成の静的サイト（ポートフォリオサイト等）を対象に、3つの観点でレビューと修正を行う手順を定義する。

## レビュー観点

### 1. セキュリティ
- フォーム入力に `maxlength` を設定し、過大なデータ送信を防ぐ（目安: name=100, email=254, message=2000）
- メールアドレス・個人情報を `mailto:` やHTMLソースに平文で書かない
  - 代替: Formspree等のフォーム中継サービスを使い、`action` 属性にエンドポイントURLのみを記載する
- 静的サイトでサーバー側レンダリングがない場合、XSSリスクは限定的だが、ユーザー入力をそのままDOMに挿入するJSコードがないか確認する
- 外部リンクに `target="_blank"` を使う場合は `rel="noopener noreferrer"` を検討する

### 2. パフォーマンス
- ファーストビュー以下に表示される `<img>` には `loading="lazy"` を付与する
- 外部ライブラリ・フレームワークを使わず素のCSS/JSで実装されているか確認する（バンドルサイズが本質的に存在しない構成を維持）
- アニメーションは `transform` / `opacity` のみを使い、レイアウト再計算を避ける（`IntersectionObserver` での遅延発火を含む）

### 3. 可読性
- TypeScript未使用の場合、型定義の代わりにCSS変数（`:root` の `--変数名`）でデザイントークンを一元管理しているか確認する
- JS は単一責任を保つ（例: `IntersectionObserver` のフェードイン処理のみを担う、フォーム送信処理と混在させない）
- セクションごとにHTMLコメント（`<!-- WORKS -->` 等）で区切り、構造を把握しやすくする

## 実施手順

1. `Read` で対象HTMLファイル全体を読み込む
2. 上記3観点に沿って表形式で現状を棚卸しする（✅問題なし / ⚠️指摘事項）
3. 指摘事項のうち実装すべきものを `Edit` で修正する
4. 修正後、`git add` → `git commit` → `git push` で本番反映する（GitHub Pages運用の場合）
5. レビュー結果と対応内容をユーザーに表形式で報告する

## 適用範囲外

- React/Vue等のフレームワークを使うプロジェクト → 別途専用のレビュー観点が必要（型定義・コンポーネント分割など）
- バックエンドAPIを持つプロジェクト → サーバー側の入力検証・SQLインジェクション対策など別観点が必要
